## 1. 요약 결론 (Executive Summary)

현재 ISDL 사이트는 "연구실의 실제 경쟁력"에 비해 "첫인상과 완성도"가 낮게 보이는 것이 가장 큰 문제다.

이번 리뉴얼은 극적인 구조 혁신보다, 다음 2가지를 정확히 달성하는 방향이 더 적합하다.

1. 예쁘고 세련되고 잘 만들어져 보이는 사이트
- 연구실의 성과와 신뢰감은 이미 충분하므로, 이를 더 보기 좋고 정돈된 형태로 보여주는 것이 핵심이다.
- 방문자가 처음 봤을 때 "오래된 연구실 사이트"가 아니라 "현재도 활발히 운영되는 연구실"로 느껴져야 한다.

2. 의뢰인이 부담 없이 관리할 수 있는 최소한의 운영 편의성
- 전체 CMS를 도입하기보다, 자주 바뀌는 일부 정보만 쉽게 수정할 수 있으면 실효성이 높다.
- GitHub 배포를 전제로, 필요한 부분에 한해 Google Sheets 또는 Google Apps Script를 연결하는 방식이 현실적이다.

핵심 결론은 다음과 같다.

1. 콘텐츠 자산은 매우 우수함
- 연구 프로젝트, 논문, 구성원 이력, 교육자료가 충분히 축적되어 있음.
- Publication 데이터는 연구실 경쟁력의 핵심 자산으로 즉시 재활용 가능.

2. 현재 문제는 콘텐츠 부족보다 "보여주는 방식"에 있음
- 전 페이지에서 VML 기반 및 오피스 내보내기 계열로 추정되는 레거시 마크업 사용.
- viewport 메타 부재, doctype 부재, 절대좌표 기반 레이아웃으로 인해 사이트가 오래되고 투박하게 보일 가능성이 높다.

3. 전면적 정보설계보다 "정돈된 시각 구성"이 우선임
- 메뉴 체계와 콘텐츠 자산은 기본 뼈대가 이미 존재한다.
- 따라서 대수술보다도, 홈/멤버/리서치/퍼블리케이션 중심으로 시각적 품질을 끌어올리는 접근이 비용 대비 효과가 크다.

4. 운영 편의성은 "모든 페이지"가 아니라 "자주 바뀌는 데이터"에 집중해야 함
- 뉴스, 멤버, 논문, 공지 등 일부 항목만 손쉽게 관리되면 체감 가치가 높다.
- GitHub Pages + 정적 사이트 + Google Sheets/Apps Script 연동이 가장 가볍고 현실적인 구성이다.

## 4. 정보구조(IA) 및 내비게이션 분석

### 4.1 현재 구조 특징

- 상단 내비게이션이 시각 오브젝트 중심으로 구성되어 의미 구조가 약함.
- research.htm, research.html 같이 URL/확장자 혼재 단서가 존재.
- 교육 콘텐츠가 education.htm과 Page2151.htm로 분절되어 탐색 맥락이 끊김.

### 4.2 주요 문제

1. 사용자 목적 기반 분류 부재
- "입학/지원", "프로젝트 협업", "논문/성과 열람", "강의자료"의 목적축이 분리되지 않음.

2. 깊이보다 단절이 문제
- 페이지 수는 많지 않으나 연결 문맥(다음 행동 유도)이 약함.

3. URL 정책 비일관
- 파일명 규칙과 확장자 혼재는 운영/SEO/유지보수 비용을 높임.

### 4.3 권장 IA(리뉴얼 기준)

- Home
- About (랩 소개, 비전, 교수 소개)
- Research (분야, 진행과제, 종료과제, 장비/인프라)
- Publications (연도, 유형, 저자, DOI 필터)
- Members (Faculty, Students, Alumni)
- News & Activities
- Education (강의, 자료실, 공지)
- Join Us (모집 안내)
- Contact (지도, 연락처, 협업문의)

해석:
- IA는 대폭 늘리기보다, 기존 메뉴 체계를 유지하면서 명칭과 우선순위만 정리하는 편이 적합하다.
- 이번 프로젝트에서 중요한 것은 페이지 수를 늘리는 것이 아니라, 방문자가 적은 클릭으로 핵심 정보를 예쁘게 보게 만드는 것이다.

## 5. UX/UI 분석

### 5.1 장점

- 콘텐츠가 실제 연구 활동과 연결되어 신뢰 신호를 제공함.
- 텍스트 기반 정보량이 충분해 데이터베이스화 가치가 높음.

### 5.2 문제점

1. 모바일 사용성 취약
- viewport 없음, 절대 좌표 배치 중심으로 작은 화면 대응 어려움.

2. 가독성/스캔성 부족
- 긴 본문과 이미지/텍스트 혼합 배치로 핵심 정보 탐색 시간이 길어짐.

3. 시각적 우선순위 미정의
- "무엇을 먼저 읽어야 하는지" 계층 구조가 약함.

4. CTA 부재
- 지원/문의/다운로드 등 전환 동작 버튼 체계가 미흡.

### 5.3 제작사 관점 개선 방향

- 카드형 섹션과 명확한 타이포 스케일로 정보 계층 재구성
- 큰 구조 개편 없이도, 여백/정렬/폰트/색상 체계를 통일해 완성도를 높임
- Home에서 30초 내 핵심 이해 가능한 요약 화면 구성
- "지원하기", "문의하기", "연구소개서 다운로드" 정도의 핵심 CTA만 명확히 노출

### 5.4 good looking 관점의 구체적 연출 포인트

1. 첫 화면에서 연구실 정체성이 바로 보이게 해야 함
- 연구실명, 한 줄 소개, 대표 연구 키워드, 대표 이미지의 조합이 필요함.

2. 논문/프로젝트/멤버를 표 대신 정돈된 카드와 리스트로 재구성해야 함
- 같은 정보라도 배치 방식만 바꿔도 훨씬 세련돼 보인다.

3. 색상과 타이포그래피가 일관되어야 함
- 중앙대학교 및 공대/연구실 분위기에 맞는 차분하고 정제된 톤이 적합하다.

4. 이미지와 숫자를 적절히 써야 함
- 최신 논문 수, 프로젝트 수, 졸업생 진출처 등은 시각적으로 요약하면 신뢰감이 커진다.

## 6. 콘텐츠 분석

### 6.1 강점 자산

1. Publication 축적량이 매우 큼
- 다년간 DOI 포함 논문 이력이 풍부하여 대외 신뢰도 확보에 유리.

2. People 데이터가 풍부함
- 재학생/졸업생/진로 정보가 연구실 브랜딩에 강력한 증거 자료.

3. 교육자료 축적
- PDF/HWP/XLS 자료가 많아 학내 교육 허브 역할 잠재력이 큼.

### 6.2 품질 이슈

1. 언어 정책 혼재
- 한국어/영어 섞임이 자연스럽지 않고 페이지별 기준이 다름.

2. 형식 불일치
- 연도 표기, 저자 표기, 링크 표기(DOI/URL) 규칙이 통일되지 않음.

3. 업데이트 신호 약함
- 일부 페이지는 최신성 인지 단서(최종 업데이트, 최신순 정렬, 태그)가 없음.

4. 파일 중심 구조의 한계
- PDF/HWP 직링크 중심은 검색성과 접근성을 떨어뜨림.

### 6.3 제작사 제안

- 콘텐츠 모델은 과도하게 세분화하지 말고 Project, Publication, Member, News 정도로 단순화
- 메타데이터 표준화: year, authors, venue, doi, status 정도의 최소 필드부터 적용
- 다국어는 전면 동시 운영보다, 우선 KR 중심 + 핵심 소개/멤버/연구만 EN 제공
- 자료실은 원본 파일 링크를 유지하되, 화면에서는 보기 좋게 목록화

## 7. 기술/표준/접근성 진단

### 7.1 기술 상태 추정

- 서버 응답 헤더: Apache-Coyote/1.1
- HTML에서 VML 네임스페이스 및 오피스 계열 스타일/클래스(Mso*) 다수 확인
- charset이 ks_c_5601-1987로 표기된 페이지 존재

이는 현대 프론트엔드 기준에서 다음 리스크를 의미한다.

1. 반응형/크로스브라우징 대응 어려움
2. 접근성 표준 준수(시맨틱/포커스/대체텍스트/명도) 취약 가능성 높음
3. 유지보수 생산성 저하(수정 난이도 높음)

### 7.2 SEO 리스크

- doctype/viewport 부재
- 구조화 데이터 부재
- 제목/설명/내부링크 전략 미흡
- 파일 링크 중심 구조로 색인 품질 저하

### 7.3 운영 리스크(정황 기반)

- 노후 구조에서 관리자 워크플로우가 수작업 의존일 가능성 높음
- 업데이트 인력 교체 시 지식 이전 어려움
- HTML 파일 직접 수정 방식은 비전공자에게 부담이 큼

### 7.4 현실적인 운영 구조 제안

- 배포는 GitHub 기반 정적 사이트로 구성
- 자주 수정되는 데이터만 Google Sheets로 관리
- 필요 시 Google Apps Script를 통해 JSON 형태로 제공하거나, 수동 내보내기 방식으로 연결

권장 운영 분리:

1. 코드로 관리할 부분
- 레이아웃, 디자인, 고정 소개 텍스트, 메뉴 구조

2. 시트로 관리할 부분
- 최신 뉴스, 재학생/졸업생 목록, 논문 리스트, 공지성 링크

이 방식의 장점:
- 디자인 완성도는 유지하면서도, 의뢰인이 자주 바꾸는 정보만 쉽게 수정 가능
- 무거운 CMS 없이도 운영 편의성을 제공할 수 있음

## 9. 핵심 개선 과제

### 9.1 반드시 즉시 반영할 과제 (P0)

1. 시각 완성도 개선
- 홈, 멤버, 연구, 논문 페이지를 중심으로 정돈된 레이아웃 적용

2. 반응형 대응
- 최소한 모바일/태블릿에서 깨지지 않고 보기 좋게 구성

3. 핵심 CTA 정리
- 문의하기, 연구소개서 다운로드, 지원 안내 정도만 명확히 노출

4. 최소 운영도구 설계
- 뉴스/논문/멤버 중 변경 빈도가 높은 항목을 Google Sheets로 관리 가능하게 구성

5. 접근성 기본 준수
- 시맨틱, 대체텍스트, 키보드 접근, 명도 대비 등 기본 수준 확보

### 9.2 고도화 과제 (P1)

1. 간단한 필터 또는 연도 구분
- 논문/뉴스를 한 번에 읽기 어렵다면 최소한의 정렬/분류 제공

2. Google Apps Script 기반 간이 관리자 연동
- 시트 입력값을 JSON 또는 정적 데이터 파일로 변환하는 흐름 구축

3. 영문 핵심 페이지 정리
- 외부 방문자 대응용 EN 버전 최소 범위 운영

4. 대표 성과 요약 시각화

## 11. 데이터 마이그레이션 및 운영 전략

### 11.1 원칙

- 기존 URL 자산 보존(가능 시 301 리디렉션)
- 핵심 문서(PDF/HWP)는 첨부 보존 + 웹 요약 본문 생성
- DOI/저자/연도 데이터 정규화

### 11.2 우선 이관 대상

1. Publication 전체
2. People(교수/재학생/졸업생)
3. Research projects (진행/종료)
4. News/Activities
5. Education 자료

### 11.3 운영 정책

- 분기별 콘텐츠 점검
- 연 1회 링크 무결성 점검
- GitHub 저장소 관리자 1명 이상 + 콘텐츠 입력 담당 1명 이상 분리

### 11.4 운영 방식 제안

안 1. 완전 정적 운영
- 내용 수정 시 저장소에서 직접 수정 후 GitHub로 배포
- 개발자 친화적이지만 의뢰인 편의성은 낮음

안 2. 부분 시트 연동 운영
- 고정 페이지는 정적으로 유지
- 뉴스, 논문, 멤버 등만 Google Sheets에서 관리
- 이번 프로젝트 목표에 가장 적합한 방식

안 3. Apps Script 보조 운영
- Google Sheets 입력값을 Apps Script로 가공해 사이트 데이터로 제공
- 편의성은 높지만, 구조를 단순하게 유지해야 관리가 쉬움

권장안:
- GitHub 정적 배포 + Google Sheets 부분 연동
- Apps Script는 정말 필요한 경우에만 최소 범위로 사용

## 14. 최종 진단 의견

현재 ISDL 홈페이지는 "내용이 부족한 사이트"가 아니라, "좋아 보이게 정리되지 않은 사이트"에 가깝다.

따라서 이번 리뉴얼의 가장 합리적인 방향은 다음과 같다.

1. 전체 구조를 과하게 흔들지 않는다.
2. 화면 완성도를 높여 세련되고 믿음직한 인상을 만든다.
3. 자주 바뀌는 일부 데이터만 쉽게 관리되게 만든다.

즉, 이번 프로젝트의 성공 기준은 "혁신적인 디지털 전환"이 아니라, 아래 두 가지다.

1. 처음 봤을 때 확실히 더 예쁘고 잘 만들어져 보일 것
2. 의뢰인이 뉴스/논문/멤버 같은 핵심 정보를 덜 부담스럽게 관리할 수 있을 것

GitHub 배포와 Google Sheets/Apps Script의 부분 활용은 이 목표에 매우 잘 맞는 조합이다.
